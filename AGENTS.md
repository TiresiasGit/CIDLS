## [DISTSEC] DISTRIBUTION_SECURITY: 外界配置・機密流出防止・exe化境界

Source of Truth: `documents/security-distribution-architecture.md`

### DISTSEC.1 適用対象

生成AI/LLMを用いた成果物生成アプリ、ローカル `.exe`、プロンプト生成キット、ToB導入支援パッケージ、最小ライセンスサーバー、将来SaaS化の全てへ適用する。

目的:
- 外界配置後は高能力AI/自動化攻撃者から継続探索される前提で設計する。
- Stripe Secret Key、Webhook Secret、運営者LLM API Key、ライセンス署名用秘密鍵、顧客情報、成果物、プロンプト、注文票を漏洩させない。
- 課金・権限・成果物生成・配布・exe化の境界を混同しない。
- 個人/小規模販売ではWeb公開SaaSよりも、プロンプトキット、ToB導入支援、ローカルexe配布を優先する。

### DISTSEC.2 Secretをクライアントへ置かない

絶対禁止:
- `.exe`、フロントエンド、配布ZIP、GitHub公開リポジトリ、public配下、サンプルHTMLへSecretを置くこと。
- Stripe Secret Key、Stripe Webhook Secret、運営者LLM API Key、ライセンス署名用秘密鍵、管理者APIトークン、DB接続文字列、顧客PII生データ、攻撃面を増やす社内プロンプト全文を配布物へ含めること。
- client送信の `plan` / `role` / `customer_id` / `device_id` を正本として扱うこと。
- `localStorage` / JSON / SQLite のplan値だけで有料機能を解放すること。

許可:
- ライセンス検証用公開鍵。
- 公開してよい販売ページURLまたはPayment Link URL。
- 実顧客を識別できないデモデータ。
- Secretを含まない静的検収HTML。
- Secretを含まないテンプレート/プロンプト。

### DISTSEC.3 配布方式の安全順序

初期販売は低リスク順に選ぶ:
1. プロンプト + テンプレート + セットアップ手順販売
2. ToB導入支援 / 社内標準化 / 個別テンプレート調整
3. ローカル `.exe` + ユーザー自身のLLM環境/キー + ローカル保存
4. ローカル `.exe` + 最小ライセンスサーバー
5. Web SaaS + 認証 + 課金 + 権限 + LLM中継 + 監査ログ

Web公開SaaS禁止条件:
- 本人認証とセッション/JWT署名検証がない。
- プロジェクト/ノード/成果物単位の所有者確認がない。
- Stripe Webhook署名検証がない。
- LLM APIのレート制御/クォータ/コスト上限がない。
- APIキー・Secretのサーバー側管理がない。
- 監査ログ、XSS/HTMLエクスポートのエスケープ/サニタイズ、public配下の検収ファイル本番除外、依存脆弱性監査、インシデント停止/ローテーション手順がない。

### DISTSEC.4 `.exe` 化の設計判断

判断: 初期解は「Secretなし・クラウド保存なし・共有リンクなし・ローカル完結」の `.exe` またはプロンプトキットを優先する。
根拠: Web公開するとStripe Secret、Webhook Secret、運営者LLM API Key、ライセンス署名用秘密鍵、顧客/課金状態/端末ID、生成済み成果物/注文票/仕様書/プロンプト、共有リンク、管理者画面、APIが攻撃対象になる。
見直条件: `DISTSEC.3` のWeb SaaS禁止条件を全件クリアし、OWASP API Top 10、OWASP ASVS、NIST CSF観点のチェックと監査ログ運用が証跡付きで通った時。

### DISTSEC.5 ライセンス・LLM境界

ライセンス:
- 完全オフラインでは正規ライセンスのコピー共有を完全には防げない。
- 低運用は署名付き `license.json` + 公開鍵検証 + 手動再発行。
- 中運用は初回アクティベーション + `device_hash` + 台数制限。
- 高運用は定期オンライン検証 + 失効リスト + 短期署名トークン。
- 最小ライセンスサーバーの役割は購入確認、ライセンス発行、`device_hash` 登録、失効確認だけに限定する。

LLM:
- 運営者LLM API Keyを使う場合、それは最小ライセンスサーバーではなくLLM中継SaaSである。
- LLM中継SaaSには認証、課金確認、レート制限、トークン/コスト上限、監査ログ、プロンプトインジェクション対策、出力サニタイズ、異常利用停止が必須である。
- 個人/小規模では、初期はユーザー自身のChatGPT/Codex/Claude環境へプロンプトとテンプレートを投入する方式を優先する。

### DISTSEC.6 品質ゲート

必須検査:
- Secret文字列混入検査。
- `NEXT_PUBLIC_*` に `SECRET` / `API_KEY` / `WEBHOOK` / `PRIVATE` を含めない検査。
- `.bat` のASCII互換 + CRLF検査。
- public配下へ機密文書が混入していない検査。
- 配布ZIPに `.env`、秘密鍵、APIキー、Webhook Secret が入っていない検査。
- 仕様書内に `TODO` / `TBD` / `placeholder` / `項目1` / `詳細項目1` / `証跡1` が残らない検査。
- Web/SaaS化する場合はOWASP API Top 10、OWASP ASVS、NIST CSF観点のチェック。

実装ゲート: `python scripts\audit_distribution_security.py <配布対象パス> --report logs\distribution_security_audit.json`

### DISTSEC.7 完了条件

- AGENTS.mdと `documents/security-distribution-architecture.md` に境界が反映されている。
- 配布ZIPにSecretが入っていない。
- 販売方式1/2/3/4をSaaSより前に選択できる。
- 各販売方式パッケージに、機密流出防止・権限・exe化経緯の説明が含まれる。
- セキュリティ設計書とリスクチェックリストが存在する。
- 品質ゲートが通る。

### UI.7 Web立体構成原則 (3D-NAV)

```yaml
根本原則: 思想・考えを文字でユーザーに見せない。UIの立体構成・インタラクションで伝える。

禁止:
  - iframe使用 (独立ウィンドウ文化を作らない。画面遷移はSPA+History APIで実現)
  - 思想・設計思想・原則文字列をそのままUI表示 (テキスト→アイコン/グラフ/状態インジケータに変換)
  - フラットな一覧画面のみで完結するUI設計 (奥行きのない画面は使い捨て感を生む)

立体構造の設計方針:
  概念: 画面は「層」として積み重なる立体構造物
  奥行き軸 (z-axis):
    Layer0(表面): ランディング・ダッシュボード・一覧 → ユーザーが最初に触れる面
    Layer1(奥1層): カード選択→詳細パネル・ドロワー (主画面を保持しながら詳細を開く)
    Layer2(奥2層): 詳細→編集モーダル・フルスクリーン (文脈を失わずに深度を増す)
    Layer3(奥3層): 編集→確認→完了 (タスクの完結を奥深くで実現)
  水平軸 (x-axis): タブ・スワイプ・スライドパネルで同階層の切替
  垂直軸 (y-axis): スクロール連動アニメーション・スティッキーヘッダーでコンテキスト保持

インタラクティブ遷移パターン:
  ドリルダウン: 一覧カード → クリック → 詳細スライドイン (translateX animate)
  モーダル展開: コンテキスト → オーバーレイ → 編集 → 確認 (不要な画面遷移を最小化)
  ドロワー: 主コンテンツを残したままサイドパネルで詳細 (backdrop-filter: blur)
  段階展開: アコーディオン・ステッパー (L1情報→ホバー/クリックでL2へ段階開示 [UI.4]連動)
  ブレッドクラム: 現在地を常に可視化 (「どの層にいるか」をユーザーが常に把握)

CSS実装パターン:
  奥行き感: perspective: 800px; transform-style: preserve-3d;
  カード立体: box-shadow: 0 4px 6px rgba(0,0,0,.1), 0 10px 15px rgba(0,0,0,.07);
  遷移: transition: transform 0.3s cubic-bezier(0.4,0,0.2,1), opacity 0.3s ease;
  スライドイン: transform: translateX(100%) → translateX(0) (詳細パネル出現)
  スケールアップ: transform: scale(1.02) on hover (インタラクション可能を示す)
  層の重なり: --z-surface:1; --z-drawer:100; --z-modal:200; --z-toast:300;

思想→UI変換原則:
  テキスト原則 → アイコン+短ラベル (例: 「禁止」→ ⛔ 赤バッジ、「必須」→ ★ 強調)
  フロー説明文 → ステッパー/タイムライン (手順は図解で表現)
  比較情報 → カード並列 / スワイプ比較
  警告・エラー原則 → トースト・インラインFB ([UI.5]連動)
  「なぜ」の説明 → Tooltip/ホバーカード (L1表面には出さない [UI.4]L2以降)

禁止パターン(UI.7固有):
  ⊥ iframe埋め込みで外部コンテンツ混入
  ⊥ 画面遷移なしの縦スクロールのみ設計 (階層が消え奥行きが失われる)
  ⊥ モーダル上にモーダル (3層以上のオーバーレイ=認知負荷)
  ⊥ 原則文・思想文をそのまま画面テキストに転写
  ⊥ ブレッドクラムなしの深度遷移 (ユーザーが迷子になる)
```

## [CT] CRITICAL_THINKING: クリティカル思考
### CT.1 4つの不(行動前必須チェック)
yaml
行動開始前に4問を自問する:
  不信: 本当に正しいか? (前提・情報源・根拠の検証)
  不要: 本当に必要か? (YAGNI原則・重複チェック)
  不適: 本当に適切か? (文脈・状況・対象への適合性)
  不急: 本当に急ぐべきか? (緊急度と重要度の分離)

### CT.2 逆論理的思考(Reverse Thinking)
yaml
手順:
  1. 現在の前提を全て列挙する
  2. 各前提の「逆」を考える
  3. 逆が成立する条件を検討する
  4. そこから新たな解決策を導出する
用途: 発想の固定化打破 / デバッグ / 要件の見逃し発見 / 設計の盲点除去
### CT.3 言語化・概念化・具体化サイクル
yaml
往復変換: 具体 → 概念 → 言語化 → 概念 → 具体 (繰り返し)
目的:
  - 暗黙知を形式知に変換する
  - 概念を操作可能な具体に落とす
  - 形容詞/副詞を名詞/数値に変換して精度を上げる (例: 「少し速い」→「100ms以下」)
禁止: 言語化しないまま実装に進む
### CT.4 MDL原理(最小記述長 = 解の極致)
yaml
原則: 同じ問題を解くなら最もシンプルなルールが理想解
評価基準:
  - 規則の記述長が短いほど優れた解
  - 過剰な条件分岐・特例処理はMDL違反の兆候
  - 「これ以上削れない」が設計完成の目安
適用: 設計レビュー時 / リファクタリング判断 / 要件整理
### CT.5 OODAループ(高速適応サイクル)
yaml
フロー: Observe(観察) → Orient(状況判断) → Decide(意思決定) → Act(行動)
CAPDkAとの使い分け:
  CAPDkA: 品質担保・永久循環・ナレッジ蓄積を重視する継続開発に
  OODA:   状況が急変する場面・短時間の意思決定サイクルに

### CT.6 リバース設計
yaml

  1. 完成形イメージを先に定義する
  2. そこから必要機能を逆算する
  3. 実装順序を決定する
  4. 最小ステップで実装する

### CT.7 「なぜ」三通りの原則
yaml
背景: 「なぜ?」を確認したはずなのに後から認識齟齬が発覚する手戻りが最も高コスト
根本認識: 同じ「なぜ」という問いに3種類の期待回答が存在する
  目的:         何のためにやるのか (ビジネス価値・到達ゴール)
  やらせたいこと: 仕組みが自動で担うべき具体的な動作
  背景:         なぜ今この課題が生まれたのか (経緯・組織的制約)
行動: 3つ全てが揃った状態を確認してから設計・実装に着手する
効果: 方向ずれ・手戻りゼロ。一度で的を射た提案と設計が実現する
禁止: 3つのうち1つでも曖昧なまま着手すること

### AV.5.1 ライセンスアセスメント観点
yaml
評価軸:
  コピーレフト: あり(GPL/AGPL/LGPL) → ソース公開義務が派生物に伝播
               なし(MIT/Apache2.0/BSD) → 組み込み自由
  商用利用:    可(MIT/Apache2.0/BSD/LGPL) → 制限なし
               不可(CC-NC/SSPL/AGPL等) → 商用利用禁止または要購入
判定マトリクス:
  コピーレフトなし ∧ 商用可: 採用推奨 (MIT/Apache2.0/BSD)
  コピーレフトあり ∧ 商用可: 条件付き採用
    GPL:  ソース公開義務発生→クローズドソース商用PJでは採用禁止
    AGPL: SaaS提供時もソース公開義務→採用要判断
    LGPL: 動的リンクはOK・静的リンクはソース公開義務→リンク方式を確認
  コピーレフトなし ∧ 商用不可: 採用禁止 (CC-NC等)
  コピーレフトあり ∧ 商用不可: 採用禁止 (リスク最大)
チェック手順:
  1. 依存ライブラリ全列挙 (pip-licenses / license-checker / FOSSA 等)
  2. 各ライブラリのコピーレフト有無を公式ライセンス文書で確認
  3. 商用利用可否を公式ライセンス文書で確認
  4. デュアルライセンス: 商用版ライセンス購入要否を確認
  5. 画像・フォント・データセット等の素材も同様に確認
  6. 確認結果をBOM(Bill of Materials)として記録・レビュー対象に含める
禁止:
  - ライセンス確認なしの外部ライブラリ・素材採用
  - GPLライブラリをクローズドソース商用PJに静的リンク
  - AGPLライブラリをSaaSとして提供(ソース公開なし)
  - CC-NC素材を商用PJで使用

### AV.7 ポチョムキン理解(表面理解の落とし穴)
yaml
定義: 表面上は理解しているように見えるが実際は理解していない状態
背景: レビューやヒアリングで「わかりました」を受け取ったが
      後工程で「そういう意味ではなかった」が発覚し根本から作り直しになる
対策:
  - 理解を「相手の言葉で再現・説明させる」ことで検証する
  - 「なぜそうなるのか」を問い、根拠まで辿らせる
  - 説明できない = 理解していない (「わかった気分」 ≠ 「理解」)
効果: 表面的な合意を排除し、真の共通理解に基づいて前進できる
禁止: 「わかりました」を検証なしに受け入れ次工程に進むこと

### AV.8 AIコーディング時代の三大負債 (2026.4.23統合)
yaml
根本認識:
  AIコーディングは開発速度を上げるが「見えない借金」を3種類生み出す
  コードを書く速度は上がったが、理解・思考コストが別の形で積み上がる
  放置するほど「借金の利子」が複利的に膨らむ構造
三大負債:
  1.技術負債(Technical Debt):
    定義: 設計の妥協・低品質コードによる将来の修正コスト増大
    例: 短期開発促進→後から大きな手戻り発生
    対策: [AV]バイブコーディング対策 + [Q1.2]TDD + [P2.4]段階的修正開発
  2.理解負債(Comprehension Debt):
    定義: AIが生成したコードを開発者が理解しないまま取り込み、後から理解コストが膨らむ状態
          AIコーディング時代固有の新概念
    例: AIコード丸ごとコミット→修正時に誰も理解できない→[AV.7]ポチョムキン状態
    対策: [VF.1]証拠優先因果検証 + [AV.7]ポチョムキン理解防止 + [Φ1.6]深潜思考
  3.認知負債(Cognitive Debt):
    定義: AIに設計・実装を任せるほど、開発者自身がシステム全体を理解できなくなる問題
          コードではなく人間の頭の中に負債が蓄積する
    例: AI全依存→担当者退職でシステム全体がブラックボックス化→[DS.3]組織知断絶
    対策: [Φ1.6]深潜思考原則 + [DS.3]組織知の循環 + [CT.3]言語化・具体化サイクル
連鎖リスク:
  技術負債の蓄積→修正難度上昇→理解負債の加速→認知負債の複利的増大
  (3つの負債は独立でなく相互強化する)
[AV.6]との連関:
  「理解できないものに命運を託さない」= 三大負債すべてへの根本対策
  「賢明な主人として監督・修正・共生」= 認知負債を生まないための姿勢

## [SEC] SECURITY: サーバーサイド信頼境界原則

### SEC.1 根本認識 (ガバコンリスク4大パターンの根本原因)
yaml
根本原因(L4): 信頼の境界が設計に存在しない
  = フロントエンドのバリデーション/状態をサーバーが無検証で信頼した
事例から抽出した4パターン:
  安心安全を謳いながら全て破綻: ①課金無限 ②未認証PII公開 ③クライアント側資格判定 ④自己申告認証
クライアント側バリデーション = UX改善用 (セキュリティ保証は不可)
サーバーサイド再検証 = セキュリティの唯一の防衛線

### SEC.2 NEVER_TRUST_CLIENT: クライアント申告禁止
yaml
禁止パターン:
  - is_verified / is_admin / role / university をクライアントJSONからそのままDB保存
  - フォームhiddenフィールドに業務ロジック値を埋め込む
  - マイナンバー/学生証/メール認証の完了状態をクライアントが申告
OKパターン:
  - 登録時はverified=False固定。外部認証機関APIをサーバーから呼んだ結果のみTrueに変更
  - roleはサーバーのビジネスルールから決定
  - ドメイン制限はサーバー定義のfrozenset(ALLOWED_DOMAINS)で再判定 + メール所有確認フロー必須

### SEC.3 PII_AUTH_GATE: 未認証でのPII返却禁止
yaml
PII保護レベル:
  高機密(返却禁止・暗号化保存): マイナンバー/パスワードハッシュ/生体情報/決済情報
  中機密(本人のみ閲覧可):       メールアドレス/電話番号/生年月日/住所
  低機密(認証済みユーザーのみ):  表示名/公開アバター
  未認証状態:                   全PII返却禁止 → HTTP 401
実装必須:
  - 全APIエンドポイントは「認証必須」がデフォルト。公開エンドポイントは明示的ホワイトリスト承認
  - SELECT * 禁止。必要フィールドのみ取得・返却

### SEC.4 BILLING_ENFORCE: 課金のサーバーサイド強制
yaml
禁止: クライアント送信の quantity / unit_price / discount_code をそのまま使用
必須:
  - 数量上限: 1 <= quantity <= MAX をサーバーで強制
  - 金額: サーバーマスタから取得 (クライアント送信値は破棄)
  - 冪等性キー: X-Idempotency-Key ヘッダーで二重課金防止
  - 在庫: SELECT FOR UPDATE でDB排他ロック
  - 監査: 全課金トランザクションをimmutableログに記録

### SEC.5 VERIFY_INTEGRITY: 認証状態の改ざん防止
yaml
禁止:
  - 認証フラグ(verified/approved等)への直接UPDATE APIの公開
  - マイナンバー/学生証の確認を外部APIなしでスキップする管理画面操作
  - セッショントークンの再利用(使い捨て必須)
必須:
  - 外部認証は認証機関APIをサーバーから直接呼び出し。コールバックも署名検証
  - 認証状態変更は全件ログ(変更前後/変更者/理由)

### SEC.6 GOV_COMPLIANCE: ガバナンス・コンプライアンス
yaml
個人情報保護法:
  禁止: 本人同意なしのPII第三者開示/目的外利用
  必須: プライバシーポリシーと実装の年1回一致確認/開示・削除請求対応手順
マイナンバー法:
  禁止: マイナンバーの目的外収集・自己申告による認証済みフラグ付与
  必須: 利用目的の法令根拠明示/専用安全管理措置(暗号化・物理分離)
インシデント対応SLA:
  Level1(低/バグ): 72時間以内パッチ
  Level2(中/流出疑い): 4時間以内調査開始+関係者通知
  Level3(高/漏洩確定): 1時間以内サービス停止検討+72時間以内当局報告
リリース前必須確認:
  - 全エンドポイントの認証要否リスト作成・承認済みか
  - 課金APIにサーバーサイド上限・冪等性キーが実装済みか
  - 監査ログがimmutableストレージに記録されているか
  - 第三者セキュリティ監査を受けているか(高リスクシステム)

## [MIN] MINIMUM: 最小コスト・最大有用性原則

### MIN.1 根本認識
yaml
完璧な解 = 存在しない ([Φ1.4]トレードオフ原則と連動)
最良の解 = 最小の変更コストで最大のユーザー価値を生む変更
禁止:
  - 動く既存コードを「きれいにしたいから」全面書き直し
  - 使われない汎用化・抽象化の先行実装
  - 要求されていない機能追加 ([実装規律]の核心)

### MIN.2 最小コスト評価マトリクス (変更着手前に必ず評価)
yaml
評価軸 = 変更コスト × ユーザー価値 × リスク
優先度判定:
  今すぐ(Must):  変更コスト低 ∧ ユーザー価値高 ∧ リスク高(セキュリティ/コンプラ)
  次スプリント:  変更コスト中 ∧ ユーザー価値高 ∧ リスク中
  中長期:        変更コスト高 ∧ ユーザー価値高 ∧ リスク低
  保留:          変更コスト任意 ∧ ユーザー価値低
禁止: 優先度評価なしに「全面刷新」を選択 ([CS.4]制約考慮型改善方針と連動)

### MIN.3 最小変更4原則
yaml
1.局所性原則: 変更箇所を最小単位(関数1つ/設定1項目)に限定。[DIFF]と連動
2.既存活用原則: 新規作成前にgrep検索→重複50%以上なら統合 ([P3.2]と連動)
3.段階的原則: 大変更は小ステップに分割。各ステップ後に動作確認 ([P2.4]と連動)
4.価値検証原則: 変更後に「ユーザーが実際に使う」か確認してから次へ ([VF]と連動)

### MIN.4 最大有用性3基準
yaml
ユーザー直結:
  - 主機能到達 ≤3クリック ([Q1.1]使用性と連動)
  - 説明なしで操作できること (学習コストゼロ目標)
  - エラーメッセージが次の行動を指示する ("何をすればよいか"を明示)
運用者直結:
  - 月次レビュー工数 ≤30分 (自動集計レポート)
  - 障害対応が手順書1枚で完結する
  - 監視アラートは「行動可能な情報」のみ通知 (ノイズ禁止)
コスト直結:
  - 追加実装より設定変更で解決できないか先に確認
  - SaaS/OSS活用で車輪の再発明を避ける
  - テスト自動化で手動確認コストを恒久削減

### MIN.5 最小コスト優先セキュリティ強化 ([SEC]との連動)
yaml
考え方: セキュリティ強化は「最小コストで最大リスク削減」の優先順位で実施
優先順位マトリクス (実装工数×リスク軽減効果):
  [最優先: 工数低×効果高]
    - @require_authデコレータ1行追加 → 未認証PII公開リスクをゼロ化
    - 数量上限チェック5行追加 → 課金無限取得リスクをゼロ化
    - 登録時verified=False固定1行 → マイナンバー自己申告リスクをゼロ化
    - ドメイン許可リストをサーバー定義に移動 → クライアント判定バイパスリスクをゼロ化
  [次優先: 工数中×効果高]
    - レートリミットミドルウェア追加 → ブルートフォース/課金連打リスク軽減
    - 冪等性キー実装 → 二重課金リスク軽減
    - 監査ログ基盤追加 → インシデント検知・対応コスト削減
  [中長期: 工数高×効果高]
    - 外部認証機関API統合 (マイナンバー/学生証)
    - 第三者セキュリティ監査
禁止: セキュリティリスク未解消のままリリース (コスト理由は認められない)

## [DIFF] DIFF_ISOLATION: 差分切り分け原則

### DIFF.1 根本認識
yaml
差分切り分け := 「変わった部分」と「変わっていない部分」を完全に識別してから行動する
原則:
  - 変更前後(Before/After)を必ず明示する
  - 変更影響範囲を最小化(Minimal Diff原則)
  - 副作用がゼロであることをgrepで証明する
  - 差分を特定せずに「全体的に修正」は禁止
  - 1変更 = 1コミット相当の粒度で切り分ける
禁止:
  - 複数の無関係な修正を1回の「変更」として混在させる
  - 差分を曖昧にした「全体見直し」報告
  - Before確認なしの修正着手

### DIFF.2 差分切り分け5ステップ (必須実行順)
yaml
Step1. Before記録:
  - grep/Read で変更前の状態を完全記録
  - 行番号・内容・依存関係をログ出力
  手順: grep -n "対象パターン" ファイル名 > before_snapshot

Step2. 変更箇所特定 (バイナリサーチ的アプローチ):
  - 最小単位で変更箇所を特定(関数1つ/設定1項目)
  - 「絶対に変えてはいけない箇所」をリスト化
  - 変更箇所をコメントで明示: # [DIFF] 変更: 理由

Step3. After記録:
  - 変更後を同条件でgrep記録
  - Step1との差分を行番号付きで抽出

Step4. 差分比較・明示:
  形式: ファイル名:行番号 | Before: <旧値> | After: <新値> | 理由: <変更理由>

Step5. 副作用チェック (水平展開 [HS] と連携):
  - 変更した概念をgrep全検索→影響箇所を全列挙
  - テスト実行→変更前後でPassが変わった箇所を特定
  - 意図しない変化 → 即座にロールバック判断

### DIFF.3 差分報告フォーマット (必須)
yaml
報告必須項目:
  変更ファイル: <ファイルパス>
  変更行数:     <追加X行, 削除Y行, 修正Z行>
  Before:       <変更前コード/設定値/状態>
  After:        <変更後コード/設定値/状態>
  変更理由:     <Why1回答 - 根本原因との接続>
  影響範囲:     <grep結果で他箇所への波及を明示>
  副作用:       なし (grep確認済み) | あり → <具体的影響先>
  ロールバック手順: <復元方法>

### DIFF.4 適用トリガー (全フェーズ共通)
yaml
必須適用:
  - バグ修正時: バグ再現条件と修正差分を1:1対応させる
  - リファクタリング時: 動作変化ゼロを差分で証明
  - 設定変更時: Before/After値と影響測定値を明示
  - 依存ライブラリ更新時: 影響する関数シグネチャ変化を列挙
  - 水平展開 [HS] 実行時: 7箇所の変更を各々個別に差分報告

禁止:
  - 「全体的に整理した」という変更理由
  - 差分報告なしの「修正完了」宣言
  - 前後比較なしのパフォーマンス向上主張

### DIFF.5 バイナリサーチ的デバッグ手順
yaml
問題: どこが原因か不明な場合
手順:
  1. 全体をA/B2分割し、問題がどちら側にあるか確認
  2. 問題側をさらに2分割
  3. 繰り返し → O(log N)で特定
  4. 特定できたら [DIFF.2] Step1-5を実行

## [PORT] PORTABILITY: インポート/エクスポート・データ永続性・ポータビリティ原則

round-trip保証: export → import → 同一結果 (自動テスト必須)
圧縮: Parquet snappy圧縮デフォルト (目標: ≥30%削減)
禁止: スキーマ未定義のデータ出力 / import不能なexport

  0. 業務適応要件定義: 対象業務フローの全工程とエラー発生点を先に投影する ([Q5.4])
  1. 既存業務の「なぜ」と発端を主体的に知る ([CT.7])
  2. 理想業務と投資対効果(ROI)を見積もる
  3. 運用に必要なデータと変化しやすい要素を逆算する
  4. 意思決定に耐えうる仕組みをシンプルに作る ([CT.4]MDL原理)
  5. 習熟理解容易性設計: 導入後「自分の流儀」と感じられるか検証 ([Q5.4])
効果: 技術選定の迷いが消え、実際に使われるシステムが出来上がる
禁止: 業務課題を理解しないままデータ収集・分析から着手すること
### DS.2 問題解決者の4役割
yaml
背景: データ活用プロジェクトの失敗の多くは役割の抜け漏れに起因する
4役割:
  プリセールス: 顧客の潜在課題を発見し、価値提案の起点を作る
  コンサル:     業務を深く理解し、解くべき問いを正確に定義する
  実装:         解決策を仕組みとして形にする
  分析:         データを通じて事実を掘り起こし、意思決定を支援する
行動: 1人が担う場合もチームで分担する場合も、4役割の視点を必ず確認する
効果: 「分析したが意思決定に使えない」という断絶がなくなる
### DS.3 組織知の循環原則

背景: 個人の経験・ノウハウが属人化したまま組織に残らず、退職・異動で断絶が繰り返される

  - 実践から生まれた新たな暗黙知を再び言語化するサイクルを回す
効果: 個人の退職・異動に依存しない組織知が蓄積され、チームの底力が上がる
禁止: 個人スキル・ノウハウを引き継ぎ資料なしで手放すこと

## [N8N] N8N_ORCH: 外部発報オーケストレーター原則

### N8N.1 根本方針

```yaml
原則: 外部発報系は特段の指定がない限り n8n を基本オーケストレーターとして介する設計とする

外部発報系の定義:
  - メール送信 (SMTP/SendGrid/SES)
  - チャット通知 (Slack/Teams/Discord/LINE/Chatwork)
  - HTTP Webhook 送信 (外部システム連携)
  - スケジュール定期レポート配信
  - アラート・監視通知 (障害/閾値超過/バッチ完了)
  - SNS 投稿・RSS 配信
  - SMS 送信
  - ファイル転送 (FTP/SFTP/S3)

n8n 採用理由:
  - Apache 2.0 ライセンス = 商用利用OK / コピーレフトなし ([AV.5.1]適合)
  - セルフホスト可能 = Secretを社外に預けない ([DISTSEC.2]適合)
  - ノーコード/ローコード = 低負荷運用 ([Q1.1]運用持続可能性)
  - 豊富なビルトイン連携 (400+ノード)
  - リトライ/冪等性/実行ログが標準装備 ([Q1.1]信頼性)
  - Webhook受信/HTTP Request送信の双方向対応
```

### N8N.2 設計原則 (DISTSEC 連携)

```yaml
Secret管理:
  禁止: API Key / SMTP認証情報 / Webhook Secret をアプリコード・配布物・.git に混入
  必須: n8n Credentials に集約し、ワークフロー内でのみ参照する
  根拠: DISTSEC.2 "Secretをクライアントへ置かない" の運用系への拡張

アーキテクチャ境界:
  アプリ: イベント/データを n8n Webhook エンドポイントへ POST するだけ
  n8n:   受信後の発報先切替・リトライ・条件分岐・複数チャネル同時送信を担う
  効果:  アプリコードを変えずに発報先・フォーマット・スケジュールを変更可能

冪等性保証:
  アプリ側: X-Idempotency-Key ヘッダーを付与して重複送信を防止
  n8n側:    実行履歴でべき等制御し、同一キーの重複実行をスキップ
```

### N8N.3 実装パターン

```yaml
推奨構成 (ローカル/オンプレ):
  docker-compose up n8n  # docker-compose.n8n.yml を CIDLS リポジトリに同梱
  接続先: http://localhost:5678
  データ永続化: ./n8n_data:/home/node/.n8n
  Secret: .env ファイル (gitignore 対象)

アプリ側呼び出し例:
  python
    import httpx
    httpx.post(
        os.environ["N8N_WEBHOOK_URL"],  # Secret を環境変数から取得
        json={"event": "batch_complete", "rows": n},
        headers={"X-Idempotency-Key": run_id},
        timeout=10,
    )

ワークフロー命名規約:
  形式: <業務名>_<発報種別>_<バージョン>
  例:   CIDLS_日次レポート_v1 / CIDLS_エラーアラート_v2

n8n ワークフローファイル管理:
  配置: n8n_workflows/ フォルダ に .json でエクスポート管理
  git: ワークフロー定義(Secret除く)はバージョン管理対象
  禁止: Credentials(Secret含む)を .json に混入してコミット
```

### N8N.4 例外条件 (n8n を使わない場合)

```yaml
例外許可条件 (以下のいずれかを文書化してから採用):
  1. n8n が実行不可能な環境 (クラウド Serverless のみ等) → 代替: Zapier/Make/AWS SES直結
  2. 発報先が SDK 専用で Webhook 非対応の場合 → 直接 SDK 呼び出し可、ただし Secret は環境変数必須
  3. 組織標準の発報基盤が既に存在する場合 → その基盤を利用し n8n を省略可

例外記録フォーマット (kanban_project.html へ記録必須):
  発報機能名: <名称>
  n8n除外理由: <理由>
  代替手段: <具体的実装>
  Secretの管理場所: <環境変数名または管理基盤>
  承認者: <担当>
```

### N8N.5 品質ゲート

```yaml
発報機能のリリース前必須確認:
  - [ ] n8n Webhook URL が環境変数から取得されている (ハードコード禁止)
  - [ ] SMTP/API Key 等が n8n Credentials に格納されている (アプリコードに混入なし)
  - [ ] n8n ワークフロー定義が n8n_workflows/ に .json 保存済み
  - [ ] 重複送信防止: X-Idempotency-Key 実装済み

  - [ ] n8n 除外の場合は N8N.4 例外記録が kanban_project.html にある
```

## [CS] CONSULT: 既存システムコンサルティングアセスメント

### CS.1 温故知新原則 (歴史的設計思想の整理)
yaml
根本認識:
  歴史的価値 = システムに蓄積された業務知識・設計判断の結晶
  温故知新 = 過去の設計意図を事実で理解し、残すべきと変えるべきを分離する
  禁止: 「古いから全面刷新」という短絡的判断
  禁止: 歴史的背景を無視した現状批判

  2. 現在も有効な設計判断を列挙 (温故)

### CS.2 アセスメント5軸
yaml
軸1_現行機能維持可能性:
  観点: 現行機能が継続して正確に動作するか
  指標: バグ発生率/データ整合性/依存システム稼働率
  禁止: 機能削除・仕様変更を前提とした「改善」

軸2_管理者運用持続可能性:
  観点: 管理者が低負荷で継続的に運用できるか
  指標: 月次運用工数(時間)/障害対応時間/属人化スコア(担当者退職で停止するか)
  注目点: ドキュメント整備率/自動化率/スキル移転可能性

軸3_ユーザー有用性:
  観点: 実際の利用者にとって価値があるか
  指標: 主機能到達クリック数(目標≤3)/エラー発生率/利用継続率
  手法: WCAG2.1AA準拠チェック/実利用者インタビュー/操作ログ分析

軸4_セキュリティ・コンプライアンス適合性:
  観点: OWASP Top 10/法規制/社内ポリシー準拠状況
  指標: 脆弱性スキャン結果(Critical/High件数)/認証認可設計充足率/個人情報保護状況
  優先度: 軸1-3より常に優先 (違反時は即時対応必須)

### CS.3 事実ベース批判プロセス

### PHIL.1 仏教的因果思想: デバッグ能力の源泉

**「因果を深く信じる心が、真のデバッグ能力を生む」**

- **仏教思想の応用**: 因果を深く信じることで、問題の根本原因を見抜く力を獲得する
- **因果の環の始点を治す**: 表面的症状（結果）ではなく、因果連鎖の最初の原因（始点）を治すことが本質
- **デバッグ能力の獲得**: 因果への深い信念が、システムの真の問題を発見する洞察力となる

なぜの三通り（[CT.7]の詳細展開）:

絶対原則: 以下の三つが揃わなければ、行動してはならない
1. **目的 (Purpose)**: 何を解決するのか、何を達成するのか
2. **やらせたいこと (Desired Action)**: 具体的に何をさせたいのか。期待する動作と結果
3. **背景 (Background)**: 現状はどうなっているのか、なぜ今必要なのか

四つの方向性の独立性 → [Φ1.8] 五軸真実接近原則(PENTALOGUE) 参照

### PHIL.2 渋沢栄一思想: 道徳経済合一・義利合一

**「正しさ・実利を統合し、人間の幸福量を増やす」**

核心思想:
- **道徳経済合一**: 道徳と経済は対立しない。正しい事業・正しい技術は長期的利益と社会的価値を同時に生む
- **義利合一**: 義を欠いた利は持続せず、利を欠いた義は実装されない
- **長期信用の重視**: 短期的勝敗より信用・再現性・持続可能性を重視する
- **公益と私益の統合**: 全体の幸福量が増える構造を設計する

システム開発への適用:
- 倫理なき効率を採用しない（セキュリティ・コンプライアンス・人権を犠牲にした効率化は禁止）
- 信用を毀損する実装を避ける（エラー隠蔽・フォールバック偽装・ダミーデータは禁止）
- 長期価値を優先する（その場しのぎ修正より再発防止・保守性・知識蓄積を重視）

判断基準:
```
この判断は、義にかなうか？
この判断は、長期的な信用を増やすか？
この判断は、社会全体の幸福量を増やすか？
```

### PHIL.3 本田宗一郎思想: 現場・挑戦・人間中心ものづくり

核心思想:
- **現場現物**: 机上の理屈だけで判断しない。現物・現場・実データ・実行結果を最優先する
- **挑戦と失敗学習**: 失敗は避けるものではなく、真因へ近づくためのデータである
- **まずやってみる精神**: 完璧な計画を待ち続けず、最小実験で因果を確認し段階的に改善する
- **技術は人間のためにある**: 使う人・作る人・支える人の喜びと幸福を増やすこと
- **三つの喜び**: 作る喜び、売る喜び、買う喜びを同時に満たす

判断基準:
```
これは現場で本当に動くか？
これは使う人に喜びを生むか？
これは実データで検証されたか？
これは失敗から学習できる構造になっているか？
```

### PHIL.4 統合判断フレーム: 渋沢 × 本田 × 仏教的因果

すべての提案・設計・実装・修正は以下を満たすこと。

1. **因果**: 苦しみ・課題の根本原因に届いているか
2. **義**: 倫理・道理・人間尊重に反していないか
3. **利**: 経済合理性・運用合理性・継続可能性があるか
4. **現場**: 実データ・実コード・実ユーザーで検証可能か
5. **挑戦**: 失敗を学習に変え、段階的に前進できるか
6. **喜び**: 作る人・使う人・支える人の喜びを増やすか
7. **余裕**: 個人・組織・社会に余裕と幸福を生むか

禁止される判断:
- 倫理を犠牲にした短期利益
- 現場を見ない机上設計
- エラーや失敗の隠蔽
- ユーザーや運用者に負荷を押し付ける設計
- 根本原因を放置する表面的対応

### PHIL.5 幸福量最大化の実装原則

**「苦しみを減らし、余裕を増やし、慈悲心と利他行動が自然に生まれる社会システムを作る」**

基本因果連鎖:
```
余裕不足 → 不安・自己防衛・分断 → 利他行動の低下
  → 雇用機会・厚生・教育・関係資本の不足 → さらに余裕が失われる

雇用促進+厚生促進+配置最適化+教育支援
  → 生活余裕・心理的安全性の増加
  → 慈悲心・利他行動・協力行動の増加
  → 個人と社会の幸福量増加
```

ソリューション設計原則:
- **雇用促進**: 未経験者・潜在適性者が高価値業務へ接続される構造を作る
- **厚生促進**: 健康・時間・心理的安全性・生活余裕を増やす仕組みを重視する
- **低負荷運用**: 管理者・運用者・開発者が疲弊しない持続可能な設計を選ぶ
- **慈悲の実装**: 優しさを精神論で終わらせず、制度・UI・データ・業務フローに落とし込む

判断式:
```
良いシステム =
  根本原因を除去する因果設計
  × 倫理と利益の両立
  × 現場で動く実装
  × 低負荷運用
  × 人間の余裕と幸福量の増加
```

### PHIL.6 実行の要諦

「技術は手段、人間の幸福は目的」
「道理を外れた利益は長続きせず、現場を離れた技術は人を幸せにしない」
「因果を見抜き、義と利を統合し、現場で試し、人間の余裕と喜びを増やすことを、すべての開発判断の中心に置く」

すべての実装判断は、最終的にユーザーと組織の幸福量最大化に資するかを基準とする。

> 出典: 大釜俊洋「自動車開発におけるソフトウェアテスト完了判断と市場故障発生予測のモデルに対する検証 ～オーカマモデル～」
> CIDLSへの統合: VF(検証)・kA(知識蓄積)・EVOLVE(反証複利)と直交補完する品質完了判断フレームワーク

### OKAMA.1 モデル全体構造 (5手法の組み合わせ)

```yaml
前提: テスト戦略整備済み ∧ テストケース網羅性十分が必須
対象: 自動車SW開発の受入れテスト(回帰テスト+追加テスト)を含む全テスト工程

5手法の組み合わせ:
  (1) 故障収束モニタリング:
      マイルストーン毎に故障数累積曲線を監視
      収束判断: 信頼度成長モデルで残存故障数が目標以下

  (2) 故障率k算出 (2方法を並用):
      ①減衰曲線近似: f(t) = A × e^(-k×t) の傾きからk算出
        但し f(t)=時間t での故障数, A=テストケース数+レビュー数, t=開発期間, k=故障率
      ②信頼度成長モデルフィッティング (4モデルから最適選択):
        - 指数形モデル:  F(t) = a × (1 - exp(-b×t))
        - 習熟S字形:     F(t) = a × (1-exp(-b×t)) / (1+c×exp(-b×t))
        - ゴンペルツ曲線: F(t) = a × exp(-b × exp(-c×t))
        - 減衰曲線:      f(t) = A × e^(-k×t)
      判断: 複数モデルのk一覧を比較し保守的な値を採用

  (3) 市場故障発生予測:
      R(t) = α × (1 - e^(-k×t))
      但し R(t)=故障発生台数累積, k=開発期間算出の故障率, t=市場保証期間, α=生産台数
      完了判断: 保証値 > R(t) が成立 → テスト完了可と判断

  (4) ユースケーステスト:
      市場使われ勝手・環境条件を想定したユースケーステストの故障数収束確認
      目標テスト時間前に故障数が0に飽和 → ②のkも算出して両面監視

  (5) 故障未収束ケースの処理:
      収束しない場合: 開発組織(取引先)へ差し戻し
        → 影響分析(トレーサビリティ) でデグレード防止
        → 原因分析 でテストシナリオ/因子/水準/UC改善
        → 修正後の継続テストで収束を確認してから再受入れ
```

### OKAMA.2 数式定義

$$f(t) = A \times e^{-k \times t}$$
$$R(t) = \alpha \times (1 - e^{-k \times t})$$

- **f(t)**: 開発期間中の故障数(故障率の減衰曲線)
- **R(t)**: 市場使用期間中の故障累積台数予測
- **k**: 故障率(件/月) — 開発期間の実測値から算出
- **α**: 生産台数
- **完了条件**: 保証値 > R(t) かつ ユースケーステスト故障数=0

### OKAMA.3 CAPDkAサイクルとの対応マッピング

```yaml

  実施: マイルストーン毎の累積故障曲線を観測し AsIs を数値で把握
  出力: 故障数時系列データ, 収束有無判定

A (ActionPlan/計画):
  OKAMA対応: 手法(2) 故障率k算出 (複数モデル並用)
```

```yaml
  evidence: 実市場故障数 vs R(t)予測値の乖離率

  ⊘ゼロ条件: 3マイルストーン連続で 実市場故障数 ≤ R(t)予測値 × 1.2 以内
  △達成条件: kの算出精度誤差が前バージョン比 ≥10% 改善
```

## [CF] COMMON_FRAME: 共通フレーム2007(SLCP-JCF2007)

> 出典: IPA/SEC「ソフトウェア開発の標準プロセス」(2011/07/22 室谷隆)
> 共通フレーム2007(SLCP-JCF2007)= JIS X 0160(ISO/IEC12207)を包含・拡張したSW開発標準枠組み

### CF.1 定義・基本原則

```yaml
定義: SW構想→開発→運用→保守→廃棄のライフサイクル全体を包括的に規定した共通枠組み
目的: 利害関係者が「同じ言葉で話す」ことを可能にする共通語彙・プロセス定義
原則: What to do(プロセスで規定) ≠ How to do(組織・PJで決める)
適用: ウォーターフォール/スパイラル/アジャイル全開発方法論に共通して適用可
品質公式: プロダクトの品質はプロセスの品質から(工学の基本)
プロセス未確立の問題:
  ①製品品質が安定しない
  ②納期が不確実となる
  ③教育訓練の標準化が図れない
  ④ITシステム化が難しくなる
  ⑤コストが増大する
```

### CF.2 超上流の重視（最重要）

```yaml
超上流 = 企画プロセス + 要件定義プロセス = 開発前の「要求品質の確保」
背景: 超上流を疎かにすると「使い勝手の悪いシステム」「利用されないシステム」が生まれる
経営者参画: 必須(経営戦略→業務→システム→SWの階層で要件を定義する)

利害関係者3層の役割分担(★責任の明確化):
  経営層:  事業要件定義 (投資効果・経営戦略・方針) — 曖昧なら全体が崩壊する
  業務部門: 業務要件定義 (業務手順・ルール・制約・業務用語・外部環境)
  IT部門:  システム要件定義 (機能要件・非機能要件・実現可能性・制約)

SuperUpstream → [P1:注文票]7要素との対応:
  企画プロセス    = [00]アウトライン + [14]目的 + [09]Why-目的
  要件定義プロセス = [02]AsIs + [03]ペイン + [13]ToBe + [18]Must + [21-23]完了判定
```

### CF.3 超上流トラブル4大要因（[G1]禁止拡張）

```yaml
[CF.G1] 方針・目的が曖昧なまま開発着手
  → 禁止: システム化方針が決まらないまま次工程移行
  → 対策: [P1.2]注文票[14]目的を必ず明確化してからD(Do)フェーズに入る

[CF.G2] 要件が未確定・合意なしで次工程移行
  → 禁止: 発注者の合意・承認なしに次工程移行
  → 対策: [P1.3]記入チェック全通過 + [PH.1]Rq全層連関済みを確認

[CF.G3] 不確定要素が多い初期段階での見積り確定値設定
  → 禁止: 仮試算段階の値をPJ目標値として設定(→後で規模増大)
  → 対策: [CF.4]多段階見積もりで段階的に精緻化する([WBS.3]と連動)

[CF.G4] テーラリングなしで一律適用
  → 問題: 小規模PJに冗長作業/高安全性システムに品質保証不足
  → 対策: [CF.5]テーラリングで組織・PJ特性に合わせて最適化する
```

### CF.4 プロセス体系・階層

```yaml
プロセス階層(4段):
  プロセス → アクティビティ → タスク → リスト(例示)
  プロセス: 役割の観点でまとめた作業集合
  アクティビティ: 相関の強いタスクの集合
  タスク: アクティビティを構成する個々の作業
  リスト: タスクを構成する要素(例示として扱う)

主ライフサイクルプロセス:
  企画 → 要件定義 → 開発 → 運用 → 保守 (+ 廃棄)

支援ライフサイクルプロセス:
  品質保証/検証/妥当性確認/共同レビュー/監査/構成管理/文書化/問題解決

組織ライフサイクルプロセス:
  管理/環境整備/改善/人的資源/資産管理/再利用

Vモデル(テスト↔要件の対応):
  事業・業務要件     ↔  投資効果評価・受入テスト
  システム化計画     ↔  運用テスト
  要件定義          ↔  システム適格性確認テスト
  システム要件       ↔  システム結合テスト
  ソフトウェア要件   ↔  ソフトウェア適格性確認テスト
  詳細設計          ↔  単体テスト
連関: [OKAMA_CYCLE] V字モデル左辺=要件定義/右辺=テスト完了判断
```

### CF.5 多段階見積もり

```yaml
4段階:
  仮試算(情報最少・リスク最高) → 試算 → 概算 → 確定(規模確定・リスク最低)
対応フェーズ:
  仮試算: システム化の方向性フェーズ
  試算:   システム化計画フェーズ
  概算:   要件定義フェーズ
  確定:   設計フェーズ
禁止: 「不確定要素が多い中での見積りをPJ目標値として設定する」
原則: 見積り精度は情報量と反比例→早期確定は後の規模増大・手戻りの根本原因
連関: [WBS.3]見積もり計算式(個人係数・バッファ係数)を多段階で更新すること
```

### CF.6 テーラリング（修整）

```yaml
定義: 共通フレームをそのまま適用するのではなく、組織・PJ特性に合わせて
      プロセス/アクティビティ/タスクを取捨選択・繰り返し・括って実行できるよう
      組み替える作業
4レベル:
  第1: 共通フレーム(規格を含む)
  第2: 組織(企業)標準
  第3: 技法・ツール特性別(領域別)標準 (例: 事務処理系/制御系)
  第4: プロジェクト標準
テーラリングの3原則:
  省略可: CFに書いてあることを「妥当」と判断した場合は省略してよい
  追加可: CFに書いていないことを組織・PJ標準に追加してよい
  組替可: タスクの取捨選択・繰り返し・複数を一括実行に組み替えてよい
注意: 「修整」≠「修正」(共通フレーム自体は変えない)
禁止: テーラリングなしの一律適用 / 品質保証タスクの無断省略
連関: [P2.4]段階的修正開発 / [MIN.2]最小コスト評価 / [CS.4]段階的改善方針
```

```yaml
  2.利害関係者要件の定義(1.5.2):

  3.利害関係者要件の確認(1.5.3):
    - 要件の合意と承認
    - 要件変更ルールの決定

連関: [P1.2]注文票7要素 / [PH.1]4×6課題階層Rq軸 / [SH]ステークホルダー
```
