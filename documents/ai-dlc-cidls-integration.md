# CIDLS AI-DLC統合方針

## 目的

CIDLSのCAPDkAを、生成AI時代の開発実務に合わせて拡張する。AIは定型生成、候補展開、WBS整形、ドキュメント初稿、コード初稿を担い、人間は問題定義、全体設計、意思決定、テストとレビュー、ユーザー価値の検収に集中する。

## Findy Team+記事から追加採用するAI-DLC詳細

AI-DLCはAI-AssistedからAI-Drivenへの転換として扱う。AI-Assistedは人間主導の工程にAIを補助として差し込む形であり、AI-DrivenはAIが質問、提案、タスク分解、初期成果物生成を主導し、人間がビジネス意図、制約、レビュー、判断、承認を担う形である。

CIDLSでは、AI-DLCを既存CAPDkAの置換ではなく、工程全体をAI前提で再設計する補助レイヤーとして扱う。SDLCやアジャイルと対立させず、反復サイクルと価値検証を残しながら、要件の明確化、レビュー設計、意思決定タイミングをAI前提に見直す。

### 10の設計原則のCIDLS解釈

| 原則 | CIDLSでの扱い |
|---|---|
| Reimagine Rather Than Retrofit | 既存プロセスにAIを後付けせず、CAPDkA内の役割分担と成果物生成順を見直す |
| Reverse the Conversation Direction | AIに質問、提案、タスク分解を促し、人間は承認、選択、修正を担う |
| Integration of Design Techniques into the Core | DDD、DA表、スイムレーン、DB設計、テスト観点を中核成果物にする |
| Align with AI Capability | AIの自律性を過信せず、人間の監督、レビュー、判断を必須にする |
| Cater to Building Complex Systems | 複雑なトレードオフ、複数関係者、運用を含むシステムで特に使う |
| Retain What Enhances Human Symbiosis | ユーザーストーリー、リスク登録簿、受入条件、レビュー会議を残す |
| Facilitate Transition Through Familiarity | 既存用語との対応を残し、1日で入りやすい運用へ落とす |
| Streamline Responsibilities for Efficiency | 役割を最小化しつつ、POと開発者の監督・検証・戦略判断責任は残す |
| Minimise Stages, Maximise Flow | ハンドオフを減らしつつ、quick-cementを避ける判断ゲートを残す |
| No Hard-Wired, Opinionated SDLC Workflows | 新規開発、改修、バグ修正、リファクタリングごとに最適フローを選ぶ |

### Mob Elaboration

Mob Elaborationは、PM/PO、開発者、QA、運用観点を持つ人がAIと同じ場で要件を具体化する作業単位として扱う。AIが「誰のための機能か」「例外ケースは何か」「非機能要求は何か」を問い、人間がその場で答え、ユーザーストーリー、受入条件、NFR、WBSへ落とす。

### Inception / Construction / Operation

| フェーズ | CIDLSでの出力 |
|---|---|
| Inception | 背景、目的、制約、優先順位、ユーザーストーリー、作業単位、前提条件 |
| Construction | 設計案、実装案、テスト案、レビュー指摘、差分、証跡 |
| Operation | デプロイ、監視、運用負荷、改善、解約・継続、ナレッジ蓄積 |

### レビュー疲れと認知負荷

AI-DLCではAIが短時間で大量の成果物を生成するため、レビュー疲れと認知負荷がボトルネックになる。CIDLSでは、重要判断だけ広いレビューに回し、局所修正は小さく処理し、UIではユーザーの脳に過剰な注意力を要求しない。

### quick-cement対策

AIが高速生成したコードや設計が早期に固まり、後で直しにくくなる状態をquick-cementとして扱う。CIDLSでは、初期生成後に必ず設計レビュー、TDD、水平展開、運用観点確認を挟み、早すぎる固定化を防ぐ。

## WBS列標準

| 列 | 目的 |
|---|---|
| タスクID | タスクを一意に追跡する |
| 機能 | 機能単位でグルーピングする |
| タスク | 動詞で完了可能な作業を表す |
| 所有者 | 責任者を明確にする |
| ステータス | Backlog / Todo / InProgress / Review / Done を管理する |
| 開始日 | 作業開始予定を明確にする |
| 終了日 | 作業完了予定を明確にする |
| 依存関係 | クリティカルパスを検出する |
| 見積(h) | 純粋な作業見積を記録する |
| 経験係数 | 担当者の経験差を反映する |
| レビュー係数 | レビュー負荷を見積へ反映する |
| 調整後(h) | 見積(h) × 経験係数 × レビュー係数 |
| 人日 | 調整後(h) ÷ 1人日の時間 |
| プロジェクトバッファ | 仕様変更、環境差、体調、レビュー集中を吸収する |
| 変更影響 | 変更時のスケジュール、スコープ、人員影響を記録する |

## 変更管理標準

変更は受け入れるが、トレードオフを必ず明示する。変更発生時は、課題管理表へ記録し、影響するWBSを特定し、再見積もりし、スケジュール延伸、スコープカット、人員変更の選択肢を提示し、合意後にproject_kanban.htmlとExcel成果物を更新する。

## レビュー負荷制御

AIは短時間で大量の成果物を出すため、人間レビューがボトルネックになる。CIDLSでは、システム全体設計、セキュリティ、DB、契約、課金、個人情報、運用停止に関わる判断は広いレビューに回し、局所的な文言修正やテスト補強は小規模レビューで処理する。

## UI立体構成と認知負荷制御

ユーザーにやさしい心が持てるぐらいの余裕が持てるように、UIは立体構造の装飾を増やすのではなく、ユーザーの脳へ過剰な注意力を要求しない形へ改良する。

Web開発の画面デザインは、原則としてiframeを使わない。画面は立体構造物として設計し、手前に現在の判断、奥に次の詳細、さらに奥に履歴や補足があるように、インタラクティブな操作と各画面を紐づける。奥深くへ画面遷移が進むイメージを、カード階層、近接配置、余白、影、フォーカス、状態遷移、パンくず、戻れる導線で表現する。

思想や考えをそのまま文字にしてユーザーに見せない。画面上では、操作のしやすさ、配置、視線誘導、状態の見え方、押せる場所、戻れる場所で伝える。WebのUIの立体構成で伝える。

CIDLSでUIを作る場合は、次を検収条件にする。

- iframeを使わない。
- ユーザーの脳へ過剰な注意力を要求しない。
- 現在位置、次の操作、戻る方法が3秒以内に分かる。
- 思想や考えをそのまま文字にしてユーザーに見せない。
- WebのUIの立体構成で伝える。
- 奥行きは情報階層と操作階層で表現し、不要な3D演出や読解負荷の高い文言で代替しない。

## 重複排除ルール

既存のCAPDkA、商用請負Excel、STORY.html、project_kanban.htmlと重なる内容は新規概念として分裂させない。AI-DLCはCAPDkAの置換ではなく、AI協働時の実行補助レイヤーとして扱う。

## ComputeUse / Computer Use 自律進化レイヤー

ComputeUseは、コード、文書、ログだけでは検証できない「画面を見て、クリックし、入力して、実際の違和感を取る」ための検証レイヤーとして扱う。OpenAIのCodex更新では、Computer UseによりCodexが画面を見てクリック・入力し、アプリやGUIを横断して作業できる方向が示されている。CIDLSではこれを、祈り入力から商用アプリを生成した後の反証実測に使う。

| 対象 | ComputeUse適用 | 代替手段 | 証跡 |
|---|---|---|---|
| Web UI | 画面遷移、入力、エラー、認知負荷の実測 | Browser Use、Playwright、スクリーンショット | PNG、ログ、テスト結果 |
| ネイティブGUI | OS設定、アプリ設定、iOS/macOSシミュレータ確認 | 手動証跡、操作手順書 | 操作録、差分、カンバン |
| 商用納品文書 | Excel、PDF、HTMLの表示崩れ確認 | openpyxl検査、HTML検査 | pytest、Excelシート検査 |
| 危険操作 | 決済、削除、送信、外部公開、契約変更は自動実行しない | 人間承認、手順書、ドライラン | 承認ログ |

自律進化の接続:
1. CIDLSパイプラインコンセプト画像から、期待するUI・成果物・出力形式を仮説化する。
2. ComputeUseまたは代替実測で、実画面の違和感、操作負荷、表示崩れ、導線不整合を取る。
3. 反証結果をproject_kanban.html、AGENTS.md、AGENTS直交圧縮.md、商用Excel、テストへ水平展開する。
4. 日次10:00の複利自己進化ループで、同じ違和感を再発させないようにルール化する。

## 祈り入力から商用アプリ成果物へ

CIDLSの最終目標は、ユーザーが自然言語で願望を述べるだけで、商用アプリ、商用請負レベル文書、A5:SQL Mk-2粒度のDB定義、テスト、運用、検収証跡までを一式生成することである。ただし、祈りをそのまま実装しない。必ず注文票、Why5、As-Is/To-Be、制約、完了判定、TDD、セキュリティ境界へ変換してから実行する。
