## TDD必須 [Q1.2]

```
Red (失敗テスト作成)
  → Green (最小限実装)
    → Refactor (品質向上)
      → 次のRedへ
```

## [SEC.Q] セキュリティ品質: 4大ガバコンリスクパターン

> 「安心安全を謳いながら設計が破綻する」実例から抽出した品質基準。
> これら全てがリリース前チェックで必須合格。

| # | リスクパターン | 根本原因 | 必須テスト |
|---|---|---|---|
| 1 | 課金アイテム無限取得 | 数量・価格のクライアント信頼 | 負数・上限超過・改ざんリクエストでサーバー拒否を確認 |
| 2 | 未ログインでPII全公開 | 認証ゲートの欠如 | curl -H "Authorization:" でPIIが0件返ることを確認 |
| 3 | クライアント側のみの資格判定 | ドメイン検証をJSに委譲 | devtoolsで制限を回避できないことをE2Eで確認 |
| 4 | 自己申告による認証状態操作 | 認証フラグのクライアント書き換え許可 | `verified=true`を含むリクエストをサーバーが拒否することを確認 |

### 品質テストシナリオ (必須)

```python
# テスト: 未認証でPIIが返らないことを確認
def test_pii_not_exposed_without_auth():
    client = TestClient(app)
    response = client.get("/api/users")
    assert response.status_code == 401  # 認証必須
    assert "email" not in str(response.json())
    assert "phone" not in str(response.json())
    assert "name" not in str(response.json())

# テスト: 課金数量の上限強制
def test_billing_quantity_server_enforced():
    response = client.post("/api/purchase",
        json={"item_id": "premium", "quantity": 999999},  # 上限超過
        headers={"Authorization": f"Bearer {valid_token}"}
    )
    assert response.status_code in (400, 422)  # サーバーが拒否

# テスト: 認証フラグの自己申告を拒否
def test_verified_flag_cannot_be_set_by_client():
    response = client.post("/api/register",
        json={"email": "test@example.com", "is_verified": True}  # 自己申告
    )
    user = get_user_from_db(response.json()["user_id"])
    assert user.is_verified is False  # サーバーがFalseに強制

# テスト: クライアント側ドメイン判定バイパス不可
def test_email_domain_validated_server_side():
    response = client.post("/api/register",
        json={"email": "hacker@gmail.com"},  # 許可外ドメイン
        headers={"X-Skip-Domain-Check": "true"}  # クライアント側バイパス試行
    )
    assert response.status_code in (400, 403)  # サーバーが拒否
```

## [UX.Q] UX品質基準 (セキュリティ × UX両立)

```yaml
エラーメッセージ品質:
  禁止 (情報漏洩につながるメッセージ):
    - "このメールアドレスは登録済みです" → ユーザー存在確認に悪用
    - "パスワードが間違っています" → IDの存在を確認される
    - "権限がありません (user_id=123 は admin でない)" → 内部情報漏洩
  必須 (統一メッセージ):
    - 認証失敗: "メールアドレスまたはパスワードが正しくありません"
    - 権限不足: "このページへのアクセス権限がありません"
    - エラー全般: "エラーが発生しました (コード: E<N>)" + サポート問い合わせ先

セキュリティ機能の可視化:
  必須:
    - 認証バッジ (メール認証済み / 学生証確認済み 等) の表示
    - プライバシー設定画面: 誰に何が見えているかを視覚的に表示
    - データ利用目的の説明 (登録フォームの各項目に「なぜ必要か」を記載)
  禁止:
    - 認証状態をユーザーが直接編集できるUI
    - 他ユーザーのメアド・電話番号がUIに露出するデザイン

説明理解容易性 (平易な言語化):
  [ ] プライバシーポリシーの冒頭に300字以内のサマリー
  [ ] セキュリティ機能 (暗号化・認証) の説明をユーザー向けに記載
  [ ] 「あなたのデータがどう使われるか」を図解で提示
  [ ] 脆弱性報告窓口 (セキュリティ@xxx.jp 等) をフッターに明示
```


## 業務適応・習熟理解容易性品質基準 [Q5.4]

> **SoT**: `AGENTS.md [Q5.4]`。定量指標・チェックリスト・3段階進化モデルの定義元。

```yaml
業務適応チェックリスト:
  - [ ] 業務フロー3項目以上をシステム内で計測可能な形で完結できるか
  - [ ] 業務エラー(転記ミス/返信漏れ/重複)が導入後に定量減少しているか (目標: 現状比>=50%削減)
  - [ ] 業務講義なしでメイン機能に到達できるか (主機能到達<=3クリック)
  - [ ] 実際の業務データで動作検証済みか (モックデータ専用検証=禁止)

習熟理解容易性チェックリスト:
  - [ ] 初回起動→主機能検索 <=5分以内で完了できるか
  - [ ] 基本機能各自忘れても再操作 1時間以内に復元できるか (定査検登録推奨)
  - [ ] 習熟層(連続1週間利用者)向けのショートカット・一括操作が存在するか
  - [ ] UIラベル・エラーメッセージが業務用語で統一されているか (IT用語メイン使用禁止)

計測指標:
  指標A: 初回利用完了までの平均ステップ数 <=3
  指標B: 1ヶ月後の継続利用率 >=80% (業務適応実現の証拠)
  指標C: 業務エラー発生底 (導入前導入後比較必須)
  指標D: ベテランユーザーの業務速度向上率 (定性アンケート数値化)

習熟3段階進化モデル:
  Stage1: 「説明を読まずに使える」 → ゼロラーニングで主機能到達
  Stage2: 「答えを読まずに使える」 → エラー時に自己解決できる
  Stage3: 「訊うことがなくなる」   → 自律的に応用・拡張できる

連鎖:
  - 習熟理解容易性と [AV.8] 理解負債は反対軸: 習熟容易なシステムが理解負債を払う
  - [TU.5] 完了判定と連動: 「受け渡し後に誰でも操作できる」=習熟完了の最小条件
  - [Phi1.2] と連動: 「システム開発 ≡ 業務フロー開発」の定動上、業務適応は第一品質上位
```
## WBS見積もり品質基準 [WBS] (定義: AGENTS.md [WBS])

```yaml
基本単位: 1人日 = 6h (PM兼任なら4h)

調整式:
  調整後(h) = 見積(h) × 個人係数 × レビューバッファ(1.2)
  人日      = 調整後(h) ÷ 6h
  全体(h)   = Σ調整後(h) × プロジェクトバッファ(1.15)

個人係数:
  1.0: 技術スタック経験あり + 類似実績あり
  1.3: 技術スタックが今回初 (学習コストあり)
  1.5: 経験浅い + 技術スタックも初
  2.0: 頻繁なサポートが必要 (OJT段階)

変更管理3択 (スコープ|スケジュール|担当):
  禁止: 「全部予定通りにやる」を選択肢にする

WBS必須列:
  タスクID / 機能 / タスク / 所有者 / 依存関係 /
  見積(h) / 個人係数 / 調整後(h) / 人日 / 開始日 / 終了日 / ステータス

AI活用方針:
  向く: 変換・可視化・テンプレート生成
  向かない: 係数設定・スコープ判断・ステークホルダー交渉
```
