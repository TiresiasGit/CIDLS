# 共通フレーム2007 (SLCP-JCF2007) 知識ベース

> 出典: IPA/SEC「ソフトウェア開発の標準プロセス」2011/07/22 室谷隆
> AGENTS.md [CF] セクションの詳細解説

## 定義と目的

共通フレーム2007 (SLCP-JCF2007) は、SW構想→開発→運用→保守→廃棄のライフサイクル全体を包括的に規定した共通枠組みです。JIS X 0160 (ISO/IEC12207) を包含・拡張しています。

目的: 利害関係者が「同じ言葉で話す」ことを可能にする共通語彙・プロセス定義

基本原則:
- What to do（プロセスで規定） ≠ How to do（組織・プロジェクトで決める）
- ウォーターフォール/スパイラル/アジャイル全開発方法論に共通して適用可
- プロダクトの品質はプロセスの品質から（工学の基本）

プロセスが確立されていない場合の問題:
1. 製品品質が安定しない
2. 納期が不確実となる
3. 教育訓練の標準化が図れない
4. ITシステム化が難しくなる
5. コストが増大する

## 超上流の重視（最重要）

超上流 = 企画プロセス + 要件定義プロセス = 開発前の「要求品質の確保」

超上流を疎かにすると「使い勝手の悪いシステム」「利用されないシステム」が生まれます。

### 利害関係者3層の役割分担

| 役割 | 担当要件 | 内容 |
|------|--------|------|
| 経営層 | 事業要件定義 | 投資効果・経営戦略・方針 |
| 業務部門 | 業務要件定義 | 業務手順・ルール・制約・業務用語・外部環境 |
| IT部門 | システム要件定義 | 機能要件・非機能要件・実現可能性・制約 |

### 注文票7要素との対応

- 企画プロセス = [00]アウトライン + [14]目的 + [09]Why-目的
- 要件定義プロセス = [02]AsIs + [03]ペイン + [13]ToBe + [18]Must + [21-23]完了判定

## 超上流トラブル4大要因

1. **方針・目的が曖昧なまま開発着手** → 対策: [P1.2]注文票[14]目的を必ず明確化してからD(Do)フェーズに入る
2. **要件が未確定・合意なしで次工程移行** → 対策: [P1.3]記入チェック全通過 + [PH.1]Rq全層連関済みを確認
3. **不確定要素が多い初期段階での見積り確定値設定** → 対策: 多段階見積もりで段階的に精緻化する([WBS.3]と連動)
4. **テーラリングなしで一律適用** → 対策: 組織・PJ特性に合わせて最適化する

## プロセス体系・階層

### 要素の階層（4段）

プロセス → アクティビティ → タスク → リスト(例示)

- プロセス: 役割の観点でまとめた作業集合
- アクティビティ: 相関の強いタスクの集合
- タスク: アクティビティを構成する個々の作業
- リスト: タスクを構成する要素(例示として扱う)

### 主ライフサイクルプロセス

企画 → 要件定義 → 開発 → 運用 → 保守 (+ 廃棄)

### 支援ライフサイクルプロセス

品質保証/検証/妥当性確認/共同レビュー/監査/構成管理/文書化/問題解決

### 組織ライフサイクルプロセス

管理/環境整備/改善/人的資源/資産管理/再利用

## Vモデル（テストと要件の対応）

| 要件レベル | テストレベル |
|---------|------------|
| 事業・業務要件 | 投資効果評価・受入テスト |
| システム化計画 | 運用テスト |
| 要件定義 | システム適格性確認テスト |
| システム要件 | システム結合テスト |
| ソフトウェア要件 | ソフトウェア適格性確認テスト |
| 詳細設計 | 単体テスト |

## 多段階見積もり

段階: 仮試算 → 試算 → 概算 → 確定（段階的に不確実性を低減）

| 段階 | 対応フェーズ | リスク |
|-----|------------|------|
| 仮試算 | システム化の方向性 | 情報最少・リスク最高 |
| 試算 | システム化計画 | 高 |
| 概算 | 要件定義 | 中 |
| 確定 | 設計 | 規模確定・リスク最低 |

**禁止**: 不確定要素が多い初期段階の見積りをPJ目標値として設定する
**原則**: 見積り精度は情報量と反比例→早期確定は後の規模増大・手戻りの根本原因

## テーラリング（修整）

定義: 組織・PJ特性に合わせてプロセス/アクティビティ/タスクを取捨選択・再構成する作業

### 4レベル

1. 共通フレーム（規格を含む）
2. 組織（企業）標準
3. 技法・ツール特性別（領域別）標準（例: 事務処理系/制御系）
4. プロジェクト標準

### テーラリングの3原則

- **省略可**: CFに書いてあることを「妥当」と判断した場合は省略してよい
- **追加可**: CFに書いていないことを組織・PJ標準に追加してよい
- **組替可**: タスクの取捨選択・繰り返し・複数を一括実行に組み替えてよい

注意: 「修整」≠「修正」（共通フレーム自体は変えない）

禁止: テーラリングなしの一律適用 / 品質保証タスクの無断省略

## 要件定義プロセス詳細（1.5）

### プロセス目的

業務・システム仕様を明確化し、利害関係者間で合意すること

### 成功成果

1. 業務・組織要件が定義・合意される
2. システム要件・制約が定義・合意される
3. 定義要件と入力ニーズとの整合性が保たれる

### アクティビティ

**1.5.1 プロセス開始の準備**
- 要件定義作業の組立て
- 利害関係者の定義と役割確認
- 要件合意・承認ルールの決定
- 要件定義環境の準備

**1.5.2 利害関係者要件の定義**
- ニーズの識別と制約定義
- 業務要件の定義（手順/入出力/組織/責任/権限/ルール/制約/業務用語/外部環境）
- 機能要件の定義
- 非機能要件の定義（★重要）
- スケジュール要件の定義
- 実現可能性とリスクの検討

**1.5.3 利害関係者要件の確認**
- 要件の合意と承認
- 要件変更ルールの決定

## CAPDkAサイクルとのマッピング

| CAPDkAフェーズ | CF対応 | 実施内容 |
|-------------|------|--------|
| C(Check) | 企画プロセス | 利害関係者3層ヒアリング / AsIs業務フロー把握 |

## 共通フレームの主な考え方（7提唱）

1. **利害関係者の役割と責任分担の明確化** → [SH]ステークホルダー / [CF.2]
2. **多段階の見積り方式** → [CF.5] / [WBS.3]
3. **Vモデルの採用** → [CF.4] / [OKAMA_CYCLE]
4. **超上流における準委任契約の採用** → [DISTSEC] / 契約境界の明確化
5. **要件の合意および変更ルールの事前確立** → [CF.7]要件定義(1.5.3)

7. **運用・保守を含めたSLCPを考えること** → [CY.1] / [CIDLS.2]Step6
