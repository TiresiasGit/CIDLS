# 既存システムコンサルティングアセスメント知識 [CS]

> **SoT参照**: 定義のSource of Truthは `AGENTS.md [CS]`。
> このファイルはdevrag RAG用実装例・コードパターンを収録。
> [PORT] アセスメント軽5軸の軽5軸目「技術的負債・将来リスク」にポータビリティ評価項目を含む。

## 概要

AGENTS.md [CS] セクションで定義する、既存システムへの事実ベース批判的アセスメント手法。
40年超の歴史を持つシステムでも「全面刷新」に逃げず、温故知新で段階的改善を実現する。

## コンサルティング最高指令 [CS.0]

> 全てのアセスメント・改善提案はこの指令に適合すること。

```
指令原文:
  「40年の歴史と設計思想を温故知新で踏まえつつ現行機能維持を前提に、
   あらゆるガバナンス・コンプライアンス違反リスク、セキュリティ、運用負荷、属人性、
   UX、説明・理解容易性の観点から事実ベースで徹底的に批判・コンサルティングアセスメント・
   将来リスク予測を行い、理想論や全面刷新に逃げず、管理者が低負荷で持続可能に運用でき、
   ユーザーにとって明確に有用で安全な状態へ強化するための、制約条件を踏まえた段階的かつ
   実行可能な改善方針と今すぐ着手すべき具体策を明示せよ。」

7観点評価軸 (全て事実ベースで評価):
  1. ガバナンス・コンプライアンス: 法規制遵守・内部統制・説明責任
  2. セキュリティ: 脆弱性・認証認可・機密保護 (OWASP Top 10)
  3. 運用負荷: 月次工数・障害対応・監視・自動化率
  4. 属人性: 担当者依存度・スキル移転可能性・引継ぎ容易性
  5. UX: ユーザー有用性・直感性・エラープルーフ (主機能到達<=3クリック)
  6. 説明・理解容易性: 習熟理解容易性・業務用語統一・ドキュメント整合性
  7. 将来リスク: 技術的負債・EOL・ビジネスリスク

禁止4項目:
  - 「一から作り直せばよい」という全面刷新提案
  - 制約条件を無視した理想的改善案
  - 「要検討」「TBD」だけで終わる報告
  - 定量根拠を示さない定性評価のみの報告

成功判定 (3条件全て充足):
  1. 管理者が低負荷で持続可能に運用できる状態への改善方針が存在するか
  2. ユーザーにとって明確に有用で安全な状態への強化策が存在するか
  3. 段階的かつ実行可能な改善方針と今すぐ着手すべき具体策が明示されているか
```

## 温故知新原則 [CS.1]

```
根本認識:
  歴史的価値 = システムに蓄積された業務知識・設計判断の結晶
  温故知新 = 過去の設計意図を事実で理解し、残すべきと変えるべきを分離
  禁止: 古いから全面刷新 / 歴史的背景を無視した批判 / 理想論への逃避

事実ベース調査4ステップ:
  1. 設計当時の制約・背景を記録から収集
  2. 現在も有効な設計判断を列挙 (温故)
  3. 時代遅れ・問題化した箇所を特定 (知新の対象)
  4. 変更コストと維持コストを定量比較
```

## アセスメント5軸 [CS.2]

```
軸1_現行機能維持可能性:
  観点: 現行機能が継続して正確に動作するか
  指標: バグ発生率 / データ整合性 / 依存システム稼働率
  禁止: 機能削除・仕様変更を前提とした「改善」

軸2_管理者運用持続可能性:
  観点: 管理者が低負荷で継続的に運用できるか
  指標: 月次運用工数(時間) / 障害対応時間 / 属人化スコア
  注目点: ドキュメント整備率 / 自動化率 / スキル移転可能性

軸3_ユーザー有用性:
  観点: 実際の利用者にとって価値があるか
  指標: 主機能到達クリック数(目標≤3) / エラー発生率 / 利用継続率
  手法: WCAG2.1AA / 実利用者インタビュー / 操作ログ分析

軸4_セキュリティ・コンプライアンス適合性:
  観点: OWASP Top 10 / 法規制 / 社内ポリシー準拠状況
  指標: 脆弱性スキャン結果(Critical/High件数) / 認証認可設計充足率
  優先度: 軸1-3より常に優先 (違反時は即時対応必須)

軸5_技術的負債・将来リスク:
  観点: 将来の変更・拡張コスト・廃止・セキュリティ劣化リスク
  指標: 循環複雑度 / テストカバレッジ(%) / 依存ライブラリEOL件数
  手法: SQALE技術的負債算定 / EOL追跡 / CVEデータベース照合
```

## 事実ベース批判プロセス [CS.3]

```
批判4段階プロセス:
  Stage1.観測事実収集:
    - ログ・エラー率・応答時間の実測値 (数値必須)
    - ユーザーフィードバック (定量化必須)
    - 脆弱性スキャン結果 (Critical/High/Medium分類)
    - 依存ライブラリEOL状況

  Stage2.構造的問題特定 (Why×5 / [PH.1] L0-L5全層):
    - 表層症状 → 根本原因の因果連鎖を解明
    - 「なぜそうなったか」の歴史的文脈を必ず保持

  Stage3.将来リスク予測 (3軸):
    - 技術的: EOL到来 / セキュリティ脆弱性の時間的拡大
    - 運用的: 担当者退職リスク / ドキュメント腐敗 / スキル断絶
    - ビジネス: 法規制強化トレンド / ユーザー離脱 / 競合優位喪失

  Stage4.改善機会の根本原因との1:1対応付け:
    - 対応なき問題指摘は禁止 (指摘した側が責任を持つ)

禁止:
  - 根拠なき「全面刷新推奨」
  - 制約条件を無視した理想論提示
  - 「要確認」「TBD」での判断先送り
  - アクションなき問題指摘だけの報告書
```

## 段階的改善方針策定 [CS.4]

```
制約前提確認 (必須・着手前に全て明確化):
  - 予算上限 (円/月・円/年)
  - 期間制約 (外部締切 / 法改正施行日等)
  - 現行機能維持要件
  - 人員・スキル制約
  - 法規制・ライセンス制約

優先度マトリクス (緊急度×インパクト):
  [今すぐ着手] 高×高: セキュリティ脆弱性/停止リスク/コンプライアンス違反
               目安: 2週間以内に着手開始
  [3-6ヶ月] 中×高: 運用効率化/主要技術的負債解消/ユーザービリティ改善
  [中長期] 低×高: アーキテクチャ近代化/テストカバレッジ/ドキュメント整備
  [保留] 低×低: 可視化のみ / 中長期レビュー時に再判断

段階的実行原則:
  - 1ステップ = 現行機能1つ不変を保証
  - 各ステップ後に [VF.1] 証拠優先因果検証を必須適用
  - ロールバック手順を事前定義 ([DIFF.3])
  - 全面リファクタリングより段階的修正を常に優先
```

### CS.3 Stage2 根本原因特定 (Why×5)

**Bug A (二重文字):**
- L1: トランスクリプトに二重文字出力
- L2: `Start-Transcript` + Python subprocess stdout の二重キャプチャ  
- L3: `& $pythonCmd` の出力を PowerShell ホストが transcript と stdout 両方に書き込む
- L4: Python `print()` が UTF-8 出力 → PS5.1 が CP932 コンソール経由で再解釈
- **対処**: `2>&1 | ForEach-Object { Write-Host $_ }` で明示的キャプチャに変更 ✅

**Bug B (日本語文字化け):**
- L1: context_latest.md に "CIDLS繝代う..." 等の文字化けが書き込まれる
- L2: `$expectedArtifacts` の日本語リテラルが PS5.1 で CP932 解釈され誤った文字列になる
- L3: ps1 ファイルの文字コード境界で UTF-8 bytes が CP932 byte解釈に化ける
- L4: PS5.1 が BOM なし UTF-8 .ps1 ファイルの日本語を CP932 として誤読
- **対処**: 日本語リテラルを `$expectedArtifacts` から除去し、`Get-ChildItem -Filter` 動的検出に変更 ✅
